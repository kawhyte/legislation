Each plan below names the model to implement it with. The axis is not severity — it's whether a wrong implementation can still pass the plan's own acceptance criteria. Grep- or build-checkable criteria ("zero references remain", "build clean") are safe for Sonnet 5, because the check catches the mistake. Criteria that need runtime judgment ("no layout jump", "1–2 requests not 6", "still looks correct") want Opus 4.8 — that's where a plan's own premise can be wrong and get checked off anyway. Haiku 4.5 isn't listed on any plan: it's for bounded side work (log triage, wide search fan-out, batch classification), not multi-file changes whose breakage shows up later.

14

AI Summary Cache Integrity

PLAN-14-critical-ai-cache-integrity.md
CRITICAL

The summarize route keys its memory cache on a 50-char title slug but Firestore on the bill id — so bills with shared title prefixes serve each other's summary, and no server-side cache read guards the free-tier token budget.

OPUS 4.8Every failure here is silent: a colliding key returns plausible-but-wrong text, and a read placed outside the dedupe still passes every test while leaking tokens under concurrency.

15

Measurement Baseline

PLAN-15-high-measurement-baseline.md
HIGH

The redesign's success metric is clicks and the app measures nothing — add five typed events before the homepage changes, so there's a control group. Vendor swap: the sink is a self-hosted Firestore collection via /api/track, not Vercel Analytics + Sentry, which are not unconditionally free.

OPUS 4.8"Silent in tests, live in prod" is a runtime-environment judgement — jsdom defines window, so a naive guard passes unit tests while polluting them.
